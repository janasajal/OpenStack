================================================================================
 LAB SCENARIO 07 — Backup and Snapshot with Retention Policy
 Topic   : VM Snapshot, Cinder Volume Backup, Automated Retention,
           and Disaster Recovery Restore Validation
================================================================================

OBJECTIVE
---------
A comprehensive backup and snapshot strategy was required to be implemented
for production VMs. Nova snapshots were configured for instance-level recovery,
Cinder volume snapshots for fast point-in-time recovery, and Cinder volume
backups for true disaster recovery to a separate backend. An automated
retention policy script was deployed via cron, and a full restore test was
performed to validate recovery procedures.

--------------------------------------------------------------------------------
BACKUP STRATEGY REFERENCE
--------------------------------------------------------------------------------

  Nova Snapshot    -> Instance-level recovery. Stored as Glance image.
                     Suitable for full VM rebuild.

  Volume Snapshot  -> Point-in-time capture at Cinder backend layer.
                     Fast recovery within same environment.
                     Lost if backend storage fails.

  Volume Backup    -> Written to separate backup backend (Swift/NFS).
                     True DR protection independent of primary backend.
                     Supports incremental backup.

STEP 5 — Automated Backup and Retention Script
--------------------------------------------------------------------------------

A retention-aware backup script was deployed for automated daily execution:

  $ cat /opt/openstack/scripts/backup-retention.sh

  #!/bin/bash
  # OpenStack VM Backup & Retention Policy Script
  # Schedule: Daily 02:00 AM via cron

  set -euo pipefail
  source /etc/openstack/admin-openrc.sh

  VM_NAME="prod-app-01"
  RETENTION_DAYS=7
  BACKUP_RETENTION_DAYS=30
  DATE=$(date +%Y%m%d-%H%M)
  LOG="/var/log/openstack/backup-${VM_NAME}-${DATE}.log"
  ALERT_EMAIL="infra-team@company.com"

  log() { echo "[$(date '+%Y-%m-%d %H:%M:%S')] $*" | tee -a "$LOG"; }

  # Nova Snapshot
  log "Starting Nova snapshot for ${VM_NAME}"
  openstack server image create \
    --name "${VM_NAME}-snap-${DATE}" --wait "${VM_NAME}" >> "$LOG" 2>&1

  # Volume Backups
  for VOL in $(openstack volume list --server "${VM_NAME}" \
    -f value -c Name); do
    openstack volume backup create \
      --name "${VOL}-backup-${DATE}" \
      --incremental --force "${VOL}" >> "$LOG" 2>&1
    log "Volume backup submitted: ${VOL}-backup-${DATE}"
  done

  # Retention - Delete old Nova snapshots
  CUTOFF=$(date -d "-${RETENTION_DAYS} days" +%Y%m%d)
  openstack image list --tag "${VM_NAME}" --tag "nova-snapshot" \
    -f value -c Name | while read SNAP_NAME; do
    SNAP_DATE=$(echo "$SNAP_NAME" | grep -oP '\d{8}')
    if [[ "$SNAP_DATE" < "$CUTOFF" ]]; then
      openstack image delete "${SNAP_NAME}"
      log "Deleted old snapshot: ${SNAP_NAME}"
    fi
  done

  # Retention - Delete old volume backups
  BACKUP_CUTOFF=$(date -d "-${BACKUP_RETENTION_DAYS} days" +%Y%m%d)
  openstack volume backup list -f value -c Name \
    | grep "${VM_NAME}" | while read BKP_NAME; do
    BKP_DATE=$(echo "$BKP_NAME" | grep -oP '\d{8}')
    if [[ "$BKP_DATE" < "$BACKUP_CUTOFF" ]]; then
      openstack volume backup delete "${BKP_NAME}"
      log "Deleted old backup: ${BKP_NAME}"
    fi
  done

  log "Backup and retention cycle complete for ${VM_NAME}"

The script was made executable and scheduled via cron:

  $ chmod +x /opt/openstack/scripts/backup-retention.sh

  $ crontab -e
  0 2 * * * /opt/openstack/scripts/backup-retention.sh \
    >> /var/log/openstack/backup-cron.log 2>&1

  $ crontab -l | grep backup
  0 2 * * * /opt/openstack/scripts/backup-retention.sh

STEP 6 — Restore Test (DR Validation)
--------------------------------------------------------------------------------

A restore test was performed in a non-production project to validate recovery.

VM was booted from the Nova snapshot:

  $ openstack server create \
      --flavor m1.medium \
      --image "prod-app-01-snap-20260518-0800" \
      --network tenant-internal-net \
      --key-name prod-keypair \
      --wait \
      prod-app-01-restore-test

  +-------------+------------------------------------------+
  | Field       | Value                                    |
  +-------------+------------------------------------------+
  | name        | prod-app-01-restore-test                 |
  | status      | ACTIVE                                   |
  | image       | prod-app-01-snap-20260518-0800           |
  | networks    | tenant-internal-net=192.168.100.50       |
  +-------------+------------------------------------------+

The data volume backup was restored to a new volume:

  $ openstack volume backup restore \
      prod-app-01-datavol-backup-20260518 \
      prod-app-01-datavol-restored

  +-------------+----------------------------------------------+
  | Field       | Value                                        |
  +-------------+----------------------------------------------+
  | backup_id   | bkp-abcd-0002-1234-5678-000000000002         |
  | volume_id   | vol-restored-abcd-1234-5678-000000000001     |
  | volume_name | prod-app-01-datavol-restored                 |
  +-------------+----------------------------------------------+

The restored volume was attached to the test VM:

  $ openstack server add volume \
      prod-app-01-restore-test \
      prod-app-01-datavol-restored \
      --device /dev/vdb

Data integrity was confirmed inside the restored VM:

  $ ssh -i ~/.ssh/prod-keypair.pem cloud-user@192.168.100.50

  [cloud-user@prod-app-01-restore-test ~]$ lsblk

  NAME   MAJ:MIN RM  SIZE RO TYPE MOUNTPOINT
  vda    252:0    0   40G  0 disk
  └─vda1 252:1    0   40G  0 part /
  vdb    252:16   0  200G  0 disk   <- restored data volume attached

  [cloud-user@prod-app-01-restore-test ~]$ sudo mount /dev/vdb1 /mnt/restore

  [cloud-user@prod-app-01-restore-test ~]$ ls /mnt/restore

  app-data  db-dumps  configs  logs   <- data confirmed intact

  [cloud-user@prod-app-01-restore-test ~]$ df -hT /mnt/restore

  Filesystem   Type  Size  Used Avail Use% Mounted on
  /dev/vdb1    xfs   200G   91G  109G  46% /mnt/restore

Test resources were cleaned up after validation:

  $ openstack server delete prod-app-01-restore-test --wait
  $ openstack volume delete prod-app-01-datavol-restored
