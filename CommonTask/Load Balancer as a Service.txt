================================================================================
 LAB SCENARIO 09 — Load Balancer as a Service (LBaaS) with Octavia
 Topic   : Production HA Load Balancer Configuration with Listener,
           Pool, Health Monitor, Floating IP, and Failover Testing
================================================================================

OBJECTIVE
---------
A production-grade load balancer was required to be configured using OpenStack
Octavia to distribute HTTPS traffic across two backend application VMs with
high availability. The configuration was built layer by layer: load balancer
object, listener, pool with session persistence, health monitor, backend
members, and floating IP association. Failover and amphora HA were validated
at the end.

STEP 5 — Pool Creation
--------------------------------------------------------------------------------

A backend pool was created with ROUND_ROBIN algorithm and HTTP_COOKIE session
persistence for stateful application compatibility:

  $ openstack loadbalancer pool create \
      --name prod-app-pool \
      --protocol HTTPS \
      --lb-algorithm ROUND_ROBIN \
      --listener prod-app-listener-443 \
      --session-persistence type=HTTP_COOKIE \
      --description "Production App Backend Pool" \
      --wait

  +----------------------+----------------------------------------------+
  | Field                | Value                                        |
  +----------------------+----------------------------------------------+
  | id                   | pool-abcd-1234-5678-efgh-000000000001        |
  | name                 | prod-app-pool                                |
  | protocol             | HTTPS                                        |
  | lb_algorithm         | ROUND_ROBIN                                  |
  | session_persistence  | {"type": "HTTP_COOKIE"}                      |
  | provisioning_status  | ACTIVE                                       |
  | operating_status     | OFFLINE                                      |
  +----------------------+----------------------------------------------+

  Algorithm selection reference:
  ROUND_ROBIN       -> Equal-weight VMs, stateless or cookie-persistent apps
  LEAST_CONNECTIONS -> Variable load, long-lived connections
  SOURCE_IP         -> Client affinity required, non-HTTP protocols

STEP 6 — Health Monitor Configuration
--------------------------------------------------------------------------------

An HTTP health monitor was configured to check the /health endpoint
every 10 seconds with a 3-failure threshold for member removal:

  $ openstack loadbalancer healthmonitor create \
      --name prod-app-healthmon \
      --type HTTP \
      --http-method GET \
      --url-path /health \
      --expected-codes 200 \
      --delay 10 \
      --timeout 5 \
      --max-retries 3 \
      --max-retries-down 3 \
      --wait \
      prod-app-pool

  +---------------------+----------------------------------------------+
  | Field               | Value                                        |
  +---------------------+----------------------------------------------+
  | id                  | hm-abcd-1234-5678-efgh-000000000001          |
  | name                | prod-app-healthmon                           |
  | type                | HTTP                                         |
  | http_method         | GET                                          |
  | url_path            | /health                                      |
  | expected_codes      | 200                                          |
  | delay               | 10                                           |
  | timeout             | 5                                            |
  | max_retries         | 3                                            |
  | max_retries_down    | 3                                            |
  | provisioning_status | ACTIVE                                       |
  | operating_status    | ONLINE                                       |
  +---------------------+----------------------------------------------+

  Health monitor logic:
  delay          = probe every 10 seconds
  timeout        = wait max 5 seconds for response
  max_retries    = 3 consecutive successes -> mark ONLINE
  max_retries_down = 3 consecutive failures -> mark ERROR, remove from pool
  expected_codes = HTTP 200 = healthy | 4xx/5xx = unhealthy

--------------------------------------------------------------------------------

Full LBaaS component status was reviewed in one command:

  $ openstack loadbalancer status show prod-app-lb

  {
    "loadbalancer": {
      "name": "prod-app-lb",
      "operating_status": "ONLINE",
      "provisioning_status": "ACTIVE",
      "listeners": [{
        "name": "prod-app-listener-443",
        "operating_status": "ONLINE",
        "pools": [{
          "name": "prod-app-pool",
          "operating_status": "ONLINE",
          "health_monitor": {
            "name": "prod-app-healthmon",
            "operating_status": "ONLINE"
          },
          "members": [
            {"name": "app-vm-01-member", "operating_status": "ONLINE"},
            {"name": "app-vm-02-member", "operating_status": "ONLINE"}
          ]
        }]
      }]
    }
  }

ROUND_ROBIN traffic distribution was verified:

  $ for i in {1..6}; do
      curl -sk https://10.50.0.150/health | python3 -m json.tool \
        | grep hostname
    done

  "hostname": "app-vm-01"   <- request 1
  "hostname": "app-vm-02"   <- request 2
  "hostname": "app-vm-01"   <- request 3
  "hostname": "app-vm-02"   <- request 4
  "hostname": "app-vm-01"   <- request 5
  "hostname": "app-vm-02"   <- request 6

  -> Perfect ROUND_ROBIN alternation confirmed.

Failover was tested by stopping the application on app-vm-01:

  $ ssh -i ~/.ssh/app-keypair.pem cloud-user@192.168.100.11 \
      "sudo systemctl stop nginx"

After 35 seconds (3 health check failures), the member was confirmed ERROR:

  $ openstack loadbalancer member list prod-app-pool

  +------------------+----------------+------+------------------+
  | name             | address        | port | operating_status |
  +------------------+----------------+------+------------------+
  | app-vm-01-member | 192.168.100.11 | 8443 | ERROR            |
  | app-vm-02-member | 192.168.100.12 | 8443 | ONLINE           |
  +------------------+----------------+------+------------------+

All traffic was confirmed routing to the healthy member only:

  $ for i in {1..4}; do
      curl -sk https://10.50.0.150/health | grep hostname
    done

  "hostname": "app-vm-02"
  "hostname": "app-vm-02"
  "hostname": "app-vm-02"
  "hostname": "app-vm-02"

Nginx was restored on app-vm-01 and member recovery was confirmed:

  $ ssh -i ~/.ssh/app-keypair.pem cloud-user@192.168.100.11 \
      "sudo systemctl start nginx"

  $ sleep 35

  $ openstack loadbalancer member list prod-app-pool

  | app-vm-01-member | 192.168.100.11 | 8443 | ONLINE |  <- recovered
  | app-vm-02-member | 192.168.100.12 | 8443 | ONLINE |

STEP 10 — Amphora HA Failover Verification
--------------------------------------------------------------------------------

Active and standby amphora roles were confirmed:

  $ openstack loadbalancer amphora list --loadbalancer prod-app-lb

  +--------------------------------------+--------+--------+
  | id                                   | status | role   |
  +--------------------------------------+--------+--------+
  | amp-abcd-0001-1234-5678-000000000001 | ALLOC  | MASTER |
  | amp-abcd-0002-1234-5678-000000000002 | ALLOC  | BACKUP |
  +--------------------------------------+--------+--------+

The active amphora was stopped to simulate a failure:

  $ openstack server stop amphora-prod-app-lb-01

Traffic continuity via the BACKUP amphora (VRRP failover) was confirmed:

  $ curl -sk https://10.50.0.150/health

  {"status":"UP","version":"2.1.0"}   <- no service interruption

Octavia automatically provisioned a replacement amphora within 2 minutes:

  $ openstack loadbalancer amphora list --loadbalancer prod-app-lb

  | amp-abcd-0003-... | ALLOC | MASTER |  <- new active amphora
  | amp-abcd-0002-... | ALLOC | BACKUP |  <- original backup promoted
