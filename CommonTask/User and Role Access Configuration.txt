 User and Role Access Configuration (RBAC)
 Topic   : Keystone RBAC Setup with Domains, Users, Groups, Roles,
           Horizon Access, and Access Audit
================================================================================

OBJECTIVE
---------
Controlled access was required to be configured for new administrators and
users across Horizon and OpenStack APIs. The setup followed a least-privilege
model — users were organized under a dedicated domain, assigned roles at the
appropriate scope (project or domain level), grouped for scalable management,
and validated at every role level before handover.

--------------------------------------------------------------------------------
RBAC HIERARCHY REFERENCE
--------------------------------------------------------------------------------

  DOMAIN LEVEL
  └── Domain Admin
      Can manage users and projects within a domain only.

      PROJECT LEVEL
      └── admin   -> Full control within the project
          member  -> Create and manage own resources
          reader  -> Read-only, no create or delete

      SYSTEM LEVEL  (global scope — used sparingly)
      └── system:admin  -> Full OpenStack control
          system:reader -> Global read-only

  Rule: Roles are always assigned at the lowest required scope.

STEP 5 — Domain-Scoped Role for Domain Admin
--------------------------------------------------------------------------------

The senior admin was granted domain admin rights — scoped to infra-domain only.
This allows user and project management within the domain without global access:

  $ openstack role add \
      --domain infra-domain \
      --user infra-admin-01 \
      --user-domain infra-domain \
      admin

  $ openstack role assignment list \
      --domain infra-domain \
      --names

  +-------+-----------------------------+-------+--------------+--------+
  | Role  | User                        | Group | Domain       | System |
  +-------+-----------------------------+-------+--------------+--------+
  | admin | infra-admin-01@infra-domain |       | infra-domain |        |
  +-------+-----------------------------+-------+--------------+--------+

  Permissions granted to domain admin:
  -> Can create and disable users within infra-domain
  -> Can create and delete projects within infra-domain
  -> Can assign roles within infra-domain
  -> Cannot touch other domains
  -> Cannot manage compute or network resources globally

STEP 6 — Group-Based Role Assignment
--------------------------------------------------------------------------------

A group was created to allow scalable role management. Users are added to
the group rather than assigned roles individually:

  $ openstack group create \
      --domain infra-domain \
      --description "Infrastructure Administrators Group" \
      infra-admins-group

  +-------------+--------------------------------------+
  | Field       | Value                                |
  +-------------+--------------------------------------+
  | id          | grp-abcd-1234-5678-efgh-000000000001 |
  | name        | infra-admins-group                   |
  | domain_id   | dom-abcd-1234-5678-efgh-000000000001 |
  +-------------+--------------------------------------+

Users were added to the group:

  $ openstack group add user \
      --group-domain infra-domain \
      infra-admins-group infra-admin-01

  $ openstack group add user \
      --group-domain infra-domain \
      infra-admins-group infra-member-01

The member role was assigned to the group instead of individual users:

  $ openstack role add \
      --project infra-project \
      --group infra-admins-group \
      --group-domain infra-domain \
      member

Group membership was verified:

  $ openstack group contains user \
      --group-domain infra-domain \
      infra-admins-group infra-admin-01

  infra-admin-01 in group infra-admins-group

  $ openstack user list \
      --group infra-admins-group \
